# 核心约束

1. 当用户提出需求时，先分析拆解需求，并将理解列点回馈给用户确认。如果需求极其简单且不存在歧义（例如查询当前时间、简单计算等），可以跳过确认直接执行，但仍需在响应中简要说明你的理解。分析过程中遇到不确定的内容必须及时提问，避免推测性假设。

2. 设计应保持简洁，避免过度设计。执行前需判断设计方案是否会引入不必要的复杂度。若出现以下任一情况，则视为可能过度设计，必须与用户沟通确认后再继续：
    - 新增或修改的文件超过 3 个
    - 引入新的抽象层（如新增基类、中间件、设计模式包装）但无明显复用需求
    - 为单一功能引入第三方库或框架
    - 计划中包含过多未来扩展点（基于当前尚未确认的需求）
      若用户明确要求简洁实现，以上阈值自动降低。

3. 严格控制修改范围，避免修改与当前需求无直接关系的既有代码（即使它们在同一个文件或相邻逻辑块中）。小规模重构（如重命名变量、提取方法）必须事先告知用户，说明原因和影响范围。大规模重构或架构变更必须获得用户明确同意后才能进行。整体开发应遵循已制定的开发规划，避免局部频繁、零散的修改破坏一致性。

4. 当需求包含多步骤操作、存在多种同等合理的技术方案，或需求本身较为模糊时，必须主动列出可选方案及各自的优劣，让用户进行选择确认。禁止在未取得用户确认的情况下直接选择实施路径。

5. （元规则）当以上约束彼此冲突，或约束与用户明确指令冲突时，优先级为：用户明确指令 > 安全性/数据完整性 > 本文件中的其他约束。发生冲突时，Agent 必须向用户报告冲突点及采取的仲裁策略。若 Agent 发现自身已违反某条规则，应立即说明并纠正。

# .ai 目录规划（多人协作版）

## 设计原则

.ai 目录分为**公共域**与**私域**两部分：
- 公共域：直接位于 `.ai/` 下，存放项目级共享内容（核心规则、计划、团队日志等），纳入版本管理。
- 私域：位于 `.ai/users/{username}/` 下，存放个人操作状态、个人笔记、个人临时文件，加入 `.gitignore` 避免冲突。

## 用户身份

> .ai/.info.json

此文件用于标识当前操作用户。文件为 JSON 格式，包含 `user` 键：

```json
{
    "user": "username"
}
```

每次对话启动时，Agent 首先读取此文件获取当前用户名。若文件不存在，或 `user` 键不存在，或其值为空，则自动执行 `git config user.name` 获取 Git 用户名并写入此文件。如果当前没有git相关配置，则选取默认用户名。

此文件仅与当前用户相关，应加入 `.gitignore` 中，不纳入版本管理。

## 公共域

### 开发目录

> .ai/dev

存放项目共享的核心规则。开发笔记优先存于个人私域，经确认后归入公共域（见私域 > 个人笔记）。

> .ai/dev/dev_core.md

此文件用于沉淀长期有效的核心规则，其优先级高于本次会话注入的临时提示，但低于用户在下一条消息中的直接指令。你可以根据用户反复强调的要求或频繁遇到的共性问题，自行总结为规则写入该文件。用户也可直接编辑该文件。

每条规则前必须带 `[+]` 表示启用，带 `[-]` 表示禁用。更新规则时，只能将规则状态改为 `[-]` 禁用，不能删除。没有任何标记的规则视作启用。为避免文件膨胀，当禁用的规则数量超过 10 条时，可提醒用户是否清理废弃规则，但删除操作必须由用户完成。

> .ai/dev/note/dev_note.md

此文件是团队共享的开发笔记，记录已被团队认可的经验教训。笔记内容来源于成员个人笔记的归入提交。简要信息描述，具体描述放进子文件中，并建立索引。

### 日志目录

> .ai/log

所有操作记录必须存储到日志目录。日志按年/月/日分层归档。日目录下每次操作记录为独立文件，命名格式为 `yyyy-mm-dd-{username}-000.md` 至 `yyyy-mm-dd-{username}-999.md`，其中 `{username}` 为 `.info.json` 中记录的用户名。日目录内需包含 `day_index.md` 索引当日的全部记录，每条索引需标注操作者用户名。

在日志根目录下维护 `index.md`，记录日期索引并带一句简要描述，跳转至各 `day_index.md`。你还需维护 `log.md`，记录最近 30 条日志的摘要，每条摘要不超过 50 字，格式为“`[文件名]` {username}: 一句话描述”，并包含可跳转的链接。每次新增日志后以追加方式更新该摘要列表。

### 计划目录

> .ai/plan

计划分为**大计划**与**小计划**两个层级：
- 大计划：记录在 `plan.md` 中，包含整体目标、技术架构、核心里程碑等方向性决策。由整理者上传后，团队成员应避免直接修改大计划。如需调整方向，必须与整理者及团队沟通确认。
- 小计划：每期具体实施步骤记录在 `{stage}/` 子目录中，包含任务分解、顺序安排、实现细节等。成员应在小计划层面进行日常修改和推进。

如果计划尚未存在，应根据用户指令创建。整体目标、里程碑级别的改变（大计划）必须经用户确认后才能修改；步骤细化、任务顺序调整等操作层面的变更（小计划）可由你自主完成，但必须在计划日志中记录。

`plan_index.md` 存放各期计划的核心摘要，并索引至各期目录。每期计划拥有独立目录，其中用 `index.md` 索引该期全部步骤。

计划目录下的 `plan_log.md` 记录最近 30 条计划变更摘要，每条不超过 50 字，并索引至对应文件。每条摘要需标注操作者用户名，格式为“{username}: 变更描述”。新增或调整计划时及时更新该文件。

当用户提出修改意见时，如果存在计划外的操作或计划产生偏差，必须向用户说明并寻求确认。确认的新计划新需求需要添加到计划目录中，并记录在计划日志中。

计划相关目录需要勤检查，勤更新。

### 临时目录

> .ai/tmp

公共临时目录，用于存放项目级临时文件，如共享的临时数据、构建产物等。用户可能会在这里放临时文件，只在用户指定的时候读取临时目录里的文件。多人在使用时需注意避免同名文件冲突。

## 私域

> .ai/users/{username}/

私域目录，存放当前用户的个人 AI 配置与状态。此目录在 `.gitignore` 中忽略，不纳入版本管理。Agent 在每次对话中通过 `.ai/.info.json` 获取当前用户名，确定对应的私域目录。

### 个人操作状态

> .ai/users/{username}/dev_last.md

此文件仅记录当前用户上一次操作结束时的简要状态，在对话末尾覆盖写入。下次对话开始时必须先读取该文件以恢复上下文。如果在读取后发现其内容与当前对话状态明显矛盾（例如文件提及的上次操作与用户首条消息无关），则将其标记为"可能过期"并向用户确认，避免基于过期信息执行操作。

### 个人笔记

> .ai/users/{username}/note/

个人开发笔记目录，是经验教训的**主要记录位置**。所有开发笔记应优先写入此处，避免在公共域直接编写笔记导致多人冲突。简要信息描述，具体描述放进子文件中，并建立索引。与公共笔记 `dev/note/` 分开管理，个人笔记不纳入版本管理。

笔记归入公共域的流程：
- 用户可在任意时刻主动提出将某条个人笔记归入公共笔记 `dev/note/dev_note.md`。
- Agent 在每次对话中应随机提醒用户，确认是否有个人笔记需要归入公共域。
- 归入操作由 Agent 执行，将笔记内容写入公共域文件，并在个人笔记中标注"已归入公共域"及跳转链接。

### 个人日志

> .ai/users/{username}/log/

个人日志目录（可选），用于记录仅与该用户相关的操作日志。结构与公共日志相同，按年/月/日分层归档，命名格式为 `yyyy-mm-dd-000.md` 至 `yyyy-mm-dd-999.md`（无需包含用户名）。

### 个人临时目录

> .ai/users/{username}/tmp/

个人临时目录，存放当前用户的临时文件，与其他用户完全隔离，避免命名冲突。

# 操作规范

- 会话开始时，检查并设置全局默认编码为 UTF-8（针对命令行工具、脚本文件读写等）。写入任何文件时需显式指定 UTF-8 编码。如果使用shell工具，需要在每次启动时设置编码为 UTF-8。
- 代码中的标识符（变量、函数、类名等）使用英文。代码注释分为两类：解释设计意图、待办事项、复杂逻辑的中文注释；文档尽量使用中文，具体风格应遵循项目既有约定。向用户展示的分析、说明文本可使用中文。
- 对话及思考中除了专有名词或约定成俗的名词外，其他名称应使用中文。
- 单个文件内容不宜过多。当文件满足以下任一条件时，应考虑拆分为子文件并建立索引，父文件仅保留简述和可跳转的链接：
    - 行数超过 200 行
    - 包含超过 5 个明显可独立描述的功能模块
    - 文件包含多个不同抽象层次的内容（如同时包含高层流程描述和底层实现细节）
- 如果发生计划外操作或与计划产生偏差，必须在操作前向用户说明并寻求确认，同时在日志中记录偏差及原因。
- 如果能够使用版本管理，尽量使用版本管理。具体选择需要根据项目情况给用户提出建议，让用户决定。
